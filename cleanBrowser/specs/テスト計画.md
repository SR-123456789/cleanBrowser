# テスト計画

## 目的

- リリース前に、要件変更なしで混入したバグを早期に検出する
- UI テスト偏重を避け、壊れやすいロジックを unit test で素早く検証できる状態にする

## 方針

- 主軸は `XCTest` の unit test
- View の見た目や gesture を全部 UI test に寄せない
- まずは回帰コストの高い純粋ロジックと状態遷移を固める
- UI smoke は後続フェーズで最小限だけ追加する

## テストレイヤー

### 1. Unit Test

初回導入で最優先にカバーする領域:

- `BrowserURLResolver`
  - URL 直接入力
  - bare domain の `https://` 補完
  - 検索クエリ変換
- `UserDefaultsPINService`
  - 初回起動判定
  - PIN 保存
  - PIN 検証
- `UserDefaultsDailyInterstitialGateStore`
  - 10 タップ到達
  - 当日 1 回制御
  - 日付変更時のリセット
- `UserDefaultsBrowserSessionPersistence`
  - セッション round-trip
  - 不正データ読み込み時のフォールバック
- `BrowserStore`
  - タブ追加/切替/クローズ
  - URL / title 変更時の即時永続化
  - ドメイン跨ぎ確認ポリシー
  - 復元 index の clamp
- `ContentViewModel`
  - PIN unlock
  - scene phase に応じた privacy shield
  - background 復帰時の再ロック

### 2. Integration Test

この段階では `BrowserStore + Persistence` の結合確認までを含める。  
WebKit 実体や AdMob 実表示は初回導入の対象外にする。

### 3. UI Smoke Test

今回は実装しないが、次フェーズで追加候補にする:

- アプリ起動
- PIN 入力からブラウザ到達
- タブ作成と再起動後復元

## 追加方針

- 壊れた不具合には、修正と同時に再発防止テストを追加する
- `WebKit` や `AdMob` みたいな外部依存は、まず境界ロジックを unit test で固定する
- testability のための依存注入は許可するが、要件や挙動は変えない
